# 第 4 课：关键字管理与性能优化

## Keywords：关键字映射表

**文件位置**: `druid/core/src/main/java/com/alibaba/druid/sql/parser/Keywords.java`

Keywords 是 Lexer 用来判断"当前标识符是不是 SQL 关键字"的查表工具。

```java
// Keywords.java:27
public class Keywords {
    private final Map<String, Token> keywords;  // 字符串 → Token
    private final long[] hashArray;             // 排序后的哈希数组
    private final Token[] tokens;               // 对应的 Token 数组

    public static final Keywords DEFAULT_KEYWORDS;  // 默认关键字集
    public static final Keywords SQLITE_KEYWORDS;    // SQLite 关键字集
    public static final Keywords DM_KEYWORDS;        // 达梦关键字集
}
```

### 双重查找机制

Keywords 提供了两种查找方式：

1. **字符串查找**（用于初始化/测试）
```java
public Token getToken(String key) {
    return keywords.get(key);
}
```

2. **哈希查找**（用于运行时）
```java
// ★ Lexer 实际使用的方法
public Token getToken(long hash) {
    int index = Arrays.binarySearch(hashArray, hash);
    if (index >= 0) {
        return tokens[index];
    }
    return null;
}
```

### 静态初始化

Druid 定义了多个静态 Keywords 实例：

```java
static {
    Map<String, Token> map = new HashMap<>();
    map.put("ALL", Token.ALL);
    map.put("ALTER", Token.ALTER);
    map.put("AND", Token.AND);
    // ... 约 200+ 关键字
    DEFAULT_KEYWORDS = new Keywords(map);
}
```

每个方言可以定义自己的 Keywords：
- `MySqlLexer.DEFAULT_MYSQL_KEYWORDS` — 增加 MySQL 特有关键字
- `SQLITE_KEYWORDS` — SQLite 特有

## SymbolTable：符号表

**文件位置**: `druid/core/src/main/java/com/alibaba/druid/sql/parser/SymbolTable.java`

SymbolTable 是 Druid 的**字符串驻留（interning）** 优化。它缓存了常见的标识符字符串，避免重复创建 String 对象。

```java
// SymbolTable.java
public class SymbolTable {
    private final String[] symbols;
    private final int mask;

    public SymbolTable(int tableSize) {
        this.mask = tableSize - 1;
        this.symbols = new String[tableSize];
    }

    public String addSymbol(String buffer, int start, int len) {
        // 1. 计算哈希
        long hash = FnvHash.fnv1a_64_lower(buffer, start, len);
        int index = (int)(hash & mask);  // 位与运算替代取模

        // 2. 查缓存
        if (symbols[index] != null && symbols[index].equals(buffer, start, len)) {
            return symbols[index];  // 命中缓存
        }

        // 3. 缓存并返回
        String str = buffer.substring(start, start + len);
        symbols[index] = str;
        return str;
    }
}
```

### 💡 为什么需要 SymbolTable？

考虑 `SELECT id, name, id FROM users`：

- "id" 出现了两次
- 没有 SymbolTable：每次都会 new String()
- 有 SymbolTable：第二次直接从缓存返回

在大量 SQL 解析的场景下（如 Druid 内置的 SQL 防火墙、监控统计），这个优化能显著减少 GC 压力。

## FNV 哈希算法

**文件位置**: `druid/core/src/main/java/com/alibaba/druid/util/FnvHash.java`

Druid 大量使用了 FNV-1a 64 位哈希算法。为什么选择它？

1. **速度快** — 只有 XOR 和乘法操作
2. **冲突率低** — 64 位哈希，适合关键字匹配
3. **实现简单** — 可以很容易地增量计算

```java
// FnvHash.java (核心算法)
public static long fnv1a_64_lower(String key) {
    long hash = 0xcbf29ce484222325L;  // FNV offset basis
    for (int i = 0; i < key.length(); i++) {
        char c = Character.toLowerCase(key.charAt(i));
        hash ^= c;
        hash *= 0x100000001b3L;       // FNV prime
    }
    return hash;
}
```

### 哈希在 Lexer 中的应用

在 `scanIdentifier()` 中，Lexer 同时计算两个哈希值：

```java
// Lexer.java:scanIdentifier 内部
long hash = FnvHash.fnv1a_64_lower(ident);  // 用于查 Keywords
long hashLCase = FnvHash.fnv1a_64_lower(ident);  // 用于识别内置函数
```

### 哈希在 Parser 中的应用

Parser 也使用哈希来快速匹配标识符：

```java
// SQLParser.java:77
protected void acceptIdentifier(Long hash) {
    if (lexer.identifierEquals(hash)) {
        lexer.nextToken();
    } else {
        setErrorEndPos(lexer.pos());
        throw new ParserException("syntax error, expect ...");
    }
}
```

这样 Parser 就不需要和字符串比较，直接比较哈希值即可。

## CharTypes：字符类型判断

**文件位置**: `druid/core/src/main/java/com/alibaba/druid/sql/parser/CharTypes.java`

这是一个工具类，提供字符类型判断方法：

```java
// CharTypes.java
public class CharTypes {
    // 判断是否为标识符起始字符
    public static boolean isIdentifierStart(char ch) {
        return Character.isJavaIdentifierStart(ch);
    }

    // 判断是否为标识符组成部分
    public static boolean isIdentifierPart(char ch) {
        return ch == '_' || ch == '$' || Character.isJavaIdentifierPart(ch);
    }

    // 判断是否为十六进制数字
    public static boolean isHex(char ch) {
        return (ch >= '0' && ch <= '9')
            || (ch >= 'a' && ch <= 'f')
            || (ch >= 'A' && ch <= 'F');
    }
}
```

这些方法被 Lexer 的 `nextToken()` 调用，用于决定如何扫描下一个 Token。

## 思考题答案

<details>
<summary>点击展开</summary>

1. **SymbolTable 的 `mask` 为什么必须是 2^n - 1？为什么要用位与运算替代取模？**
   - 2^n - 1 在二进制下是全 1（如 2^10-1 = 1023 = 0b1111111111）。
   - `hash & mask` 等价于 `hash % (mask+1)`，但位与运算比取模快得多（CPU 1 个时钟周期 vs 几十个周期）。
   - 这要求表大小是 2 的幂，这也是 `SymbolTable(512)` 使用 512（2^9）的原因。

2. **FNV 哈希相比 MD5/SHA 有什么优缺点？**
   - 优点：极快（只有 XOR + 乘法），适合**运行时频繁计算**的场景；实现简单（20 行代码）。
   - 缺点：不是加密哈希，可以碰撞构造；哈希空间 64 位，理论上碰撞概率比 MD5(128位)/SHA(256位) 高。
   - Druid 的场景是关键字匹配，碰撞概率极低（几千个关键字 vs 2^64 空间），FNV 是**性价比最高**的选择。

3. **如果两个不同的关键字 FNV 哈希冲突了，Druid 会怎么处理？**
   - `Keywords` 的构造函数 `new Keywords(map)` 内部使用 `TreeMap<Long, String>`，如果两个关键字哈希相同，后面的会覆盖前面的。但 FNV-1a 64 位在几百个关键字的规模下碰撞概率极低。
   - 如果要更严谨，Druid 应该在初始化时检测哈希冲突并抛异常。目前的做法是"假设不会碰撞"。
</details>
